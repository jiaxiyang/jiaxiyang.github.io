NOTE

  1. 这是因为链接时（link time）写死的 RPATH 或 RUNPATH 在 .so 文件中被保存了下来，即使运行时没有设置 LD_LIBRARY_PATH，动态链接器（ld.so）也能根据 .so 里的这些路径去找到对应的库。
    • readelf -d /usr/local/lib/python3.10/dist-packages/torch_sipu/lib/libtorch_sipu.so | grep -i 'rpath\|runpath' 查看动态库rpath和runpath
  2. readelf --debug-dump=decodedline build/cmake.linux-x86_64-cpython-3.10/bin/triton-tensor-layout | grep -oE '/[^ ]+\.cpp:' | sort -u 获取文件中debug信息包含的C++文件
  4. 使用 -H（GCC/Clang）查看头文件包含的顺序。用于定位头文件顺序引起的问题
  5. (good)log 太多可以只把 error 重定向到文件 command 2> error_log.txt
  6. std::numeric_limits<float>::max() float 最大值
  7. -Wno-xxx 不启动 xxx warning
  8. gcc 编译阶段打印宏定义的内容
  9. 100-gcc-tips
  10. 没 link 报undefined reference to, link 但没 found xxx was not found, cannot find xxx
  11. target_link_options(xxx PUBLIC -Wl,-unresolved-symbols=ignore-in-shared-libs) 只要能找到 xxx 中的符号就行， link 的 shared lib 中未找到符号不报错， 忽略 link 的库解析，用于交叉编译，运行时能找到库
  12. dbg-macro
    • 灵感来源rust !dbg
  13. c++23 支持 stack trace basic_stacktrace
    • backward-cpp
    • cpptrace
    • boost stack trace
  14. 二进制依赖时， 需要先 find_package(glog REQUIRED), 否则
    • op PRIVATE 依赖 glog: undefined reference to 错误, 未传递出 glog 依赖 （静态链接的可以 PRIVATE 依赖)
    • op PUBLIC 依赖 glog: target was not found 错误, 传递出 glog 依赖
  15. gcc/g++ --verbose显示编译过程
    • set(CMAKE_CXX_FLAGS "--verbose") or -DCMAKE_CXX_FLAGS="--verbose" cmake 也可以显示编译信息； 比 -DCMAKE_VERBOSE_MAKEFILE=ON 显示更详细
  16. strip 是一个在 Unix-like 操作系统中用于删除或剥离目标文件或对象文件的符号信息的工具。
    • 删除符号表信息：符号表包含了源代码中的函数和变量的名称。
    • 删除调试信息：debug section 被删除
  17. function tracing 只需要 symbols，不需要 debug symbols（gcc -g）。strip 之后不能跟踪
  18. 动态符号（.dynsym）vs. 局部符号（.symtab）， .symtab 包含.dynsym; 可以用命令 strip 来手动去掉局部符号表， strip 前后用readelf -s ./test看, strip 只去除了.symtab
  19. strip 之后 gdb 不能使用 function 名字加断点
  20. strip 保留 symbol 文件，gdb 可使用 symbol-file 命令加载 symbol
  1. debug info 作用：
    • 将内存地址映射到具体某行源代码
    • 调用栈展开（stack unwinding）
  2. 在 C++中，如果函数有指定返回类型，但在实际代码中没有返回值或者在所有路径上都没有返回值，那么它的行为是未定义的（Undefined Behavior）。根据 C++标准，未定义的行为可能导致任何结果，包括程序崩溃、生成错误的结果或者在某些情况下看起来像是“正常工作”。
  3. addr2line其功能是将函数地址解析成文件名或行号的工具。 addr2line debug
  4. 查看 debug 模式：gdb l and info source
  5. -DCMAKE_BUILD_TYPE=RelWithDebInfo release 模式编译，带 debug info
    • 优化：开启（通常 -O2 或 -O3），以提高运行效率。 调试信息：仍然保留调试符号（-g），但由于优化的影响，变量和代码结构可能会与源代码对不上。
    • 和Debug的区别：link
  6. -DCMAKE_CXX_FLAGS="-g" cmake 编译选项添加 debug info； release 模式也适用
  7. gcc 选项顺序很重要，如果两个选项冲突，则以后一个为准。可以用-O3 -fno-inline-functions既使用-O3 的功能又关闭函数内嵌功能。
