重构

  1. 最重要的是重构的节奏感，小的步子可以更快的前进，请保持代码永远处于可工作状态。（chunye 就是这样)
  2. 重构前现有测试
  3. 小步修改
  4. 每次修改后就运行测试
  5. 每次重构就提交代码，push 前把修改合并成更有意义的提交
  6. 永远将函数的返回值命名为”result”
  7. 尽量移除局部变量，使用小函数替代（临时变量会带来麻烦）
  8. 重构的关键在于运用大量微小且保持软件行为的步骤，一步步达成大规模修改。重构过程中几乎可以随时停下来，不影响软件功能。
  9. 区分重构和性能优化；重构是为了让代码更容易理解，更容易改变。程序可能更快，也可能更慢。在性能优化是，只关心让程序运行的更快，代码可能更难理解和维护。
  10. 两顶帽子：添加新功能和重构。添加新功能时不应该修改既有代码，只管添加新功能，重构时不能添加新功能，只管调整代码结构
  11. 重构的主要功能：
    • 改进软件设计
    • 使软件更容易理解
    • 帮助找到bug
    • 提高编程速度
  12. 重构的唯一目的就是让我们开发更快，用更少的工作量创造更大的价值。
  13. 代码所有权边界：接口的使用者（用户）与声明者（作者）彼此独立，声明者无权修改使用者的代码。添加新接口，旧的接口标记为不推荐使用(deprecated)。接口会变得复杂。
  14. 重构和性能：有性能测试工具，发现最耗时代码进行优化

Principles of Software Engineering

  1. 迭代和重构。
  2. 自动化
  3. 可重复
  4. 学习性测试：使用第三方代码，第一步不要在生产代码中实验新东西，而是单独编写测试来浏览和理解第三方代码。区分学习和整合。
  5. 小步快跑，节奏
  6. KISS (Keep It Simple Stupid)
  7. DRY (Don’t Repeat Yourself) (generic programing, 提出重复代码公用….)
  8. 层次(抽象接口组成):写代码思路不应该是树状，而应该是有层次的，抽象出来层次，屏蔽细节，只依赖上个层次，不关注更低级别的层次, 每一层都相当于一种语言，解决一类问题，这样的系统更健壮。见sicp
  9. 抽象,控制复杂度， 黑盒抽象
  10. 边界，隔离
  11. 具体规则：
    • 命名规则：类名是名词，方法名是动词。
    • 函数规则：尽可能短小，只做一件事，每个函数一个抽象层次，名字不怕长，尽量少参数，无 side effect.
    • 类规则：短小，单一权责，高内聚(方法和变量互相依赖）
    • 系统设计规则： 多写测试，不可重复，表达力(好名字，好结构…), 尽可能少的类和函数(和前三条冲突，优先级最低）
  12. 一键： 单个命令构建系统，单个命令运行所有测试
  13. 出错仔细看 log，调试步步为营
  14. 多看文档
  15. 记录工作日志
  16. 区分对内和对外，区别对待
  17. 不要重复造轮子
  18. YAGNI (You Aren’t Gonna Need It) link
  19. 高内聚，低耦合
  20. 正交
  21. 分离控制和逻辑。控制：与业务逻辑无关的代码或系统的控制，如：多线程，异步，部署等，逻辑：业务逻辑，即解决用户问题的逻辑。
  22. 权衡
  23. 实用易用
  24. SOLID 原则
    • 单一职责原则 SRP 一个类只做一件事
    • 开闭原则 OCP 软件中的对象（类，模块，函数等等）应该对于扩展是开放的，但是对于修改是封闭的
    • 里氏替换原则 LSP 子类应该可以完全替代父类，也就是说在使用继承时，只扩展新功能，不要破坏父类原有的功能。
    • 接口隔离原则 ISP 客户端不应该依赖它不需要的接口。如果一个接口在实现时，部分方法由于冗余被客户端空实现，则应该将该接口拆分，让实现类只依赖自己需要的接口。
    • 依赖倒置原则 DIP 细节应该依赖与抽象，抽象不应该依赖于细节。把抽象层放在程序设计的高层，并保持稳定，程序的细节变化由低层的实现层来完成。

思维模式

  1. 官方资料 first
  2. 3W: what, why, how dds introduction about
    • 查看各种技术先看 about
  3. 5W2H: 原因（为什么 Why）、对象（是什么 What）、地点（在哪里 Where）、时间（什么时候 When）、人员（是谁 Who）、方法（怎么做 How）、程度（How much）
  4. 可重复和自动化是非常重要的思维工作方式。版本管理 git, Jenkins, docker, CI, Rust Cargo.lock 文件, emacs tmux 插件, 软件一条命令安装(自动化，可重复), 代码风格 fmt 是自动化也是风格可重复，work log(可重复）。
  5. effect和side effect副作用。纯函数的行为只依赖它的接口。非纯函数副作用：引起环境，上下文的改变。side effect 指的是那些任何主动或被动地对“非当前局部环境变量”的访问的操作。注意，“非当前局部环境变量”不仅仅包括访问全局变量和静态变量。这个操作可能非常广泛，甚至不局限于变量，还包括访问外部 IO，执行某些特殊指令，以非正常返回的方式引起的主动或被动出栈（比如异常）等行为。
  6. 多用纯函数，非纯函数会增加系统状态，使系统复杂化，容易出问题。组合爆炸
  7. monad

  65. 语言feature: 编程语言有一堆 feature，每种语言在其中选择权衡，理解了各种 feature，也就学会了各种语言，各 feature 之间的关系，互斥，相交。haskell feature haskell book

  66. 函数式编程没有变量也没有 side effect。隔离了可变性，变量或者默认状态默认就是不可变的。避免了死锁，状态冲突等众多麻烦。

  75. 将知识点放入体系中，不能孤立的学习。比如学习 C++知识时要有big picture，要把编译器，操作系统考虑进去，这样才能理解深刻，不要陷入细节中无法自拔。

  76. 搭建自己编程体系，哲学。写代码是实践，用来验证体系的好坏。重点应该关注体系哲学而不是具体的代码。

  77. 将知识点带入现实，自己融入进去。便于理解和讲解。比如，想象自己是操作系统，管理一堆资源，向应用提供服务，交互过程。

  78. 如何看待编程？编程是一个建造世界的过程(会涉及不同层级的抽象)，建造完成后的世界就相当于程序，完成了编程的目标。建造流程为生产对象，组织对象，对象行为约定。（对应设计模式里的创建型模式，结构型模式，行为型模式），设计模式就相当与搭建过程中用到的各种套路，是一种设计思路。参考《我的世界》游戏。

  79. 区分抽象层次很重要，有助于理解系统的横向(同层次）纵向（上下层）关系。理解编写代码要从最高层次（顶层设计）入手，一层一层的往里剥，越往里越细节（the onion principle）。

  80. 有目标或者带着问题看文档或代码。

  81. 无论是整体还是模块，都要先弄清楚模块目的，再学习基本概念, 然后学习框架和设计模式，最后再学习具体代码。

  82. 看文档或代码的时候抓关键点记笔记，有利于专注，记录的时候会强化，加深理解，也有利于新思路涌现。可以记录到blog或者git上。如有需要进行可视化

  83. 看文档代码先问why。要从模块存在的目的？解决了什么问题？特点特性？优缺点？适用范围？概念？架构？设计思路？具体实现方式？等方面考虑问题。

  84. 概念刚开始可能很难理解，可以从具体实例来理解，多动手，写一遍理解深刻很多。

  85. 找到语言优势，重点关注。C++的优势在于性能，死磕性能。

  86. 学习建模，对系统建模。

  87. 读代码时要先理解总体设计架构，设计思路，再具体到细节，架构理解了代码就不是问题。

  88. 多学点架构，设计方法，少学点语法。

  89. 最终目标创造而不是复制。

  90. 架构是撑起项目的基础，理解架构了就能看清楚系统轮廓。架构就相当于系统的草图，有了草图才能起高楼。

  91. 注意软件架构和设计模式的区别。

  92. 设计是解决问题的方案，不是一种算法。

  93. 分主题分模块学习语言，分拆效率高，更容易 focus，不能一股脑的啥都往脑子里塞。

  94. 学习新知识时先明确概念，如 cy 学习 git，明确 git 的原理，四个 object；学习 cmake 时明确 component 和 target；学习 gstreamer 明确 component,pad 等，给别人讲解是也是从基本概念、基本组成讲起。

  95. 文档越详细越好，肯定会忘

  96. 读代码的时候可以记录自己的理解，可加深印象。用 git 仓库保存起来。

  97. github 排名，github trending

  98. 查看开发者的 github 找资料

  99. 先模仿github 开源代码。youtube 视频教程。

  100. 随时写一些简单的例子验证想法。很重要。

  101. Github awesome 系列，awesome emacs, shell，rust

  102. emacs org 管理项目，todo，日志

  103. 遇见问题记得看 github issues

  104. 设计的思路，设计原则，理解之后对使用帮助很大(例如： emacs, git)

  105. 编译时出错仔细看错误信息

  106. 调试时要一小步一小步的调，步步为营

  107. 乐于学习新东西，学习 emacs 之后才知道 vim 该怎么用，学习 Rust 之后才知道语言一些基本特性，更好的理解 C++

  108. 重头文件(接口)，轻源文件。看一个新库时例如 xir, vart，先看接口（接口注释很重要），知道接口功能就行，不需要明白具体实现。通过接口明白整个库的结构。

  109. 突出重点:读音加重，停顿。看 C++代码要具备忽略次要信息的能力，看重点的信息要用不同的眼光，例如函数名，函数功能重点看和理解，具体实现可以先不用管

  110. 程序有 Debug 和 Release 模式，Release 开优化，编译慢，运行快

  111. 程序有编译期和运行期。

  112. 程序包含功能和性能，注重分析程序的性能，比如帧率，时间

  113. 流程: 开发(开发新库），测试(jenkins)，发布(上 github)，运营(gihub issues)

  114. 重设计，轻实现

  115. 按部就班，不能急于求成

  116. 知乎话题：rust, emacs

  117. 学习编程语言要关注几点：起因，解决什么问题，设计哲学，特性 参考文章

  118. 大项目中，模块之间要检查输入输出是否正确，不用在乎模块里内部实现

  119. 日志写在实现的外面，可以从使用者的角度检查输出，日志写在里面，可以从实现者的角度检查问题。倾向写在实现外面，先定位哪个模块出问题了，再深入到模块内部，看看为什么出问题。

  120. 一个函数完成一个小功能，方便理解定位问题

  121. main.rs 函数中处理程序运行，lib.rs 中处理任务逻辑

  122. 操作符重载打印结构体很方便

  123. API 文档目的是让他人理解如何使用库，而不是让他人明白库如何实现的。

  124. 创建一个有用的公有 API 结构更像是一门艺术而非科学，你可以反复检视他们来找出最适合用户的 API

  125. 留意 FAQ

  126. 函数(一层抽象)，类，泛型都是为了重复代码。

  127. 写代码时明确（输入，输出，算法）（开，关，做）（构造，析构，虚函数)

  128. RFC 流程。

  129. C++ API -> C API 可能拆分， C API -> Rust API 可能合并

  130. 区分公开与非公开部分。

  131. 学习语言，应该了解编译器，编译器是整个编程中重要一环。

  132. 使用 cppinsights 从编译器的角度看程序。有助于对语法的了解。

  133. 区分计算密集和 I/O 密集，计算密集多线程，I/O 密集异步？

  1. 在物理学中，第一性原理(或起源)，或称从头算，指从基本的物理学定律出发，不外加假设与经验拟合的推导与计算。例如利用薛定谔方程在一些近似方法下解电子结构，但不从实验数据得到拟合参数的从头计算法。
  2. LLVM 之父 Chris Lattner：这正是我们从第一性原理出发开展工作的原因。你必须从头完成所有的工作，如果做得对，就不应跳过任何重要的步骤 link
  3. Any problem in computer science can be solved by another layer of indirection 计算机科学领域的任何问题都可以通过增加一个间接的中间层来解决;例如虚拟地址
  4. Think twice, code once
  5. Hackers write better code when it's for their own satisfaction instead of for pay.

  7. Any fool can write code that a computer can understand. Good programmers write code that humans can understand. –Martin Fowler
    任何一个傻瓜都会写能够让机器理解的代码，只有好的程序员才能写出人类可以理解的代码。——Martin Fowler
  8. “The first 90% of the code accounts for the first 90% of the development time. The remaining 10% of the code accounts for the other 90% of the development time.” – Tom Cargill
    “最开始的 90%的代码使用了程序员 90%的时间，剩下的 10%的代码也需要 90%的开发时间”——Tom Cargill（这不就是中国谚语——“行百步半九十”）
  9. 过早的优化是万恶之源。Premature optimization is the root of all evil! - Donald Knuth
